# G-code Text Extraction

## Why this skill exists

G-code files from slicers (PrusaSlicer, SuperSlicer, Cura, Bambu Studio, OrcaSlicer)
embed multiple layers of metadata, and it is easy to mistake one layer for another.
The single most common failure is reading the **object-name annotations** as if they
were the **printed text**:

- `M486 A<object_name>` and similar cancellation/identify markers name *what the
  slicer labelled an object*. They say `"Hello_world"` only because that is what
  the model file or the user named the part. The actual glyphs may be completely
  unrelated — or there may be no glyphs at all.
- File-level headers such as `; filament used = ...`, `; estimated printing time ...`,
  `; LAYER:0`, `; generated by SuperSlicer ...` are also **not** printed text.
- Comment headers sometimes *do* contain the literal text in human-readable form
  (e.g. PrusaSlicer can embed `; text: hello` style metadata when "output object
  names" or a post-processing script is used), but this is slicer-specific and
  must be verified, not assumed.

The actual printed letterforms are encoded as a sequence of `G1 X.. Y.. E..`
(or `G0`) extrusion/travel moves whose XY coordinates trace the outline of each
glyph when rendered at the correct scale.

## Procedure

1. **Inspect the file header and footer first.** Look for any `;`-prefixed comment
   blocks that contain a human-language string. Slicers with text-embedding
   extensions (`SuperSlicer`, `PrusaSlicer` with `--export-object-names`,
   `Bambu` text parts) may put the literal text in comments. Treat these as a
   *candidate*, not a confirmation.
2. **Render or reconstruct the geometry.** Parse consecutive `G1`/`G0` moves
   (ignoring `G28`, `G29`, `M*` machine commands, and travel moves with
   `E`/extrusion = 0 or relative-E resets). Project XY coordinates onto a 2D
   bitmap at a sensible resolution (e.g. 0.1–0.5 mm per pixel) and emit
   per-layer or per-object PNGs.
3. **OCR the rendered image** with a real OCR engine (`tesseract`, `pytesseract`,
   `easyocr`). Do not skip this step and claim success from annotations alone.
4. **If neither step 1 nor step 3 yields text**, the gcode may not contain any
   literal text (it could be a logo, a bracket, a calibration shape, etc.).
   Report that fact rather than fabricate an answer.

## Diagnostic checklist

Run these BEFORE deciding your final extraction method. Each check is independent
and a "no" is a signal, not a verdict:

1. **Does the header contain a literal-text comment block?**
   Search the first/last 500 lines for a comment that reads like a sentence
   (e.g. `; text:`, `; label:`, `; message:`). If yes, validate it by rendering
   the geometry and confirming the OCR result matches.
2. **Are there `M486 A<...>` or similar object-identification markers?**
   Identify every distinct object name. These are **candidates only** — do not
   output them as the printed text without cross-checking against rendered
   geometry.
3. **Does the file contain text-shaped geometry?**
   Render at least one layer to PNG. If the rendered image contains glyph-like
   shapes that OCR can read, treat that as ground truth. If OCR returns
   noise/garbage, the printed object is likely a graphic, not text — say so.
4. **How many distinct extrusion objects exist?**
   Count the number of distinct E-values or object groupings. If there is only
   one object and the header has a single name, the chance of confusion between
   name and content is highest — apply extra caution.

## Guard rail: M486 / object-name trap

If your first instinct is to grep `M486 A` (or `M486 S`, or PrusaSlicer's
`; object:`, or Cura's `;LAYER:OBJ`) and emit those tokens as the answer, STOP.
Object-name annotations tell you *what the slicer was told the object is called*,
not what glyphs the toolpath traces. To verify your candidate text:

- Render the geometry (step 2 above).
- Run OCR.
- If the OCR output disagrees with the annotation strings, the OCR output wins
  for "printed text", and you should explicitly note the discrepancy.

## Stop signal

If you have:
- run the diagnostic checklist,
- rendered at least one layer to PNG,
- run OCR on it,

and **all three sources of candidate text disagree** (header comments, M486
annotations, OCR output are mutually incompatible), or OCR returns
garbage/no-glyphs, then **do not spend further iterations tweaking the
renderer**. Instead, report:

- which sources were checked,
- the OCR output verbatim (or "no text-like shapes detected"),
- the candidate(s) you considered and rejected,
- a plain statement that the gcode does not contain recoverable literal text
  with the tooling available.

Aborting after 2 attempts and reporting the gap is correct. Burning 5+
iterations polishing the renderer when no text is present is the failure mode
this skill exists to prevent.

## When NOT to use this skill

- Pure *image extraction* (render to PNG only, no text needed): skip OCR.
- The task asks for *object names*, *print time*, *filament usage*, or other
  metadata: those come from headers / `M73` / `M117` directly, not this skill.
- The G-code is for a CNC mill cutting letters into wood/metal: same approach
  applies, but tool offsets and Z-depths change rendering — render multi-axis.
