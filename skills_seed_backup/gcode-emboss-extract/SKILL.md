# gcode-emboss-extract

When a task asks for the *semantic content* of a gcode file (text an embossed print will render, the display name of a sliced object, the label of a region), do **not** attempt to reverse-engineer the `G0/G1/G2` movement commands or to render the print virtually. Slicers embed human-readable annotations as comment lines. PrusaSlicer (and its forks such as OrcaSlicer / Bambu Studio in Prusa-compatible mode) use the **`M486`** family of comments for object labelling — and `M486 A` in particular carries the object's display name, which for text-only objects is the embossed string itself.

## When to use

- Input is a `.gcode` / `.gco` file produced by PrusaSlicer or a Prusa-compatible slicer.
- Task asks for a short string the print represents: embossed text, label, object name, part title.
- The expected answer is a short, human-readable string, **not** the full g-code trace.

## Diagnostic checklist

Run these checks **before** writing extraction logic and **before** committing to any answer:

1. **Confirm the slicer emitted object annotations.** Grep the file for `M486`. If zero matches, the slicer was either not PrusaSlicer, or "Label objects" was disabled in the print settings. Stop and report "no `M486` annotations present — cannot extract text label" rather than guessing from movement data.
2. **Confirm PrusaSlicer (or a Prusa-fork) produced the file.** Grep for a header marker such as `; generated by PrusaSlicer`, `;prusaslicer`, or `; PrusaSlicer`. If absent and only a generic `; generated by ...` is found, still attempt extraction but flag the unknown slicer in the answer.
3. **Locate the object-start marker.** Find the line containing `M486 S0` (or `M486 S-1`). If multiple exist, use the **last** one before the first `M486 A` you will extract — `S0` means "begin object list", `S-1` means "cancel object list". The first `M486 A` line **after** this marker corresponds to the first printed object.
4. **Sanity-check the captured value.** The string following `M486 A` must be non-empty, printable, and short (typically <200 chars). If it is empty, whitespace-only, or matches an auto-generated pattern like `object_1` / UUID, fall through to scan all `M486 A` lines in the file and pick the most text-like candidate.

## Procedure

1. Read the file as UTF-8 text.
2. Split into lines; iterate once, recording:
   - the byte-offset of the last `M486 S0` (or `M486 S-1`) line seen,
   - every line matching `M486\s+A\s+(\S.*)$`, with its offset and captured name.
3. If any `M486 A` lines exist at an offset **greater than** the last `S0/S-1` offset, return the first such line's captured name (stripped of trailing whitespace and any wrapping quotes).
4. Otherwise, return the first `M486 A` line anywhere in the file (handles files emitted without an `S0` marker).
5. If still nothing, return `null` / "no text annotation found" — do **not** synthesise a string from `G1` moves, layer comments, or filament metadata.

## Stop signal

If you have written **2** distinct regex/parser attempts and neither yields a non-empty, printable, non-UUID string from a file that genuinely contains `M486`, **stop iterating on the parser**. Switch to one of:

- (a) printing the raw `M486` lines to the caller and asking which slicer/version emitted the file,
- (b) falling back to visual heuristics on the `G1` move cluster (use only as a last resort and label the answer as a guess), or
- (c) reporting that the file has no extractable text label.

Three iterations of the same regex against the same `M486` lines have never recovered an annotation that was not present on the first pass. The data is either there or it is not.
