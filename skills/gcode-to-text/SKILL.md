# G-code to Text

Translate G-code into a clear, structured natural-language description of what the machine will actually do. The output is for a human reader (operator, reviewer, documentation), not for a machine.

## When to use

Trigger on requests like:
- "Explain / describe / summarize this G-code"
- "Convert this G-code to plain English / text / human-readable form"
- "Annotate / comment / document this .gcode file"
- "What does this toolpath / program do?"
- "Read this .nc / .ngc / .tap / .cnc / .gcode file and tell me what it does"

Input forms: a file path, a pasted block of G-code, or a directory containing G-code files. If the user supplies a directory or multiple files, ask which to process unless they said "all".

## Output contract

Produce a Markdown document with these sections, in order. Omit a section only if there is genuinely nothing to report for it.

1. **Summary** — 2–4 sentences. What kind of program is this (3D print, milling job, laser cut, probing routine, macro), the target machine flavor if detectable (Marlin / RepRap / Klipper / GRBL / LinuxCNC / Fanuc / Haas / Mach3), and the high-level intent (e.g. "5-pass facing operation on a 100×80 mm stock", "PLA print of a 42 mm tall part, 0.2 mm layers").
2. **Setup & modal state** — units (G20 inch / G21 mm), positioning mode (G90 absolute / G91 relative), plane (G17/G18/G19), workpiece coordinate system (G54–G59), tool number(s), spindle/laser/extruder state, coolant, feedrate units (G94/G95).
3. **Bounding box & travel** — X/Y/Z min–max actually visited, total commanded travel distance if computable, max feedrate seen, max spindle/laser power seen. Mark estimates as estimates.
4. **Step-by-step narrative** — chronological prose walk-through, grouped into logical phases (Startup, Tool change, Operation 1, …, Shutdown). Each phase: a heading, then 1–5 sentences describing what happens and why it appears to happen. Reference line numbers (`L123`) when pointing at specific commands.
5. **Notable commands & risks** — anything an operator should double-check: missing homing, hard-coded absolute positions without a WCS, rapid moves through stock, dwell loops, M0/M1/M6 stops, raw spindle commands without warmup, unsupported codes for the detected flavor, comments that contradict the motion.
6. **Unknown / vendor-specific codes** — any G/M code you could not confidently interpret, with the line number and your best guess.

Keep the prose concrete. Prefer "rapids to X120 Y0 at Z safe height" over "moves the tool". Quote the actual numbers from the file.

## Method

1. **Read the input.** If it's a file, read it; if it's pasted, work from the message. For files over a few thousand lines, sample: read the first ~200 lines (setup), last ~100 lines (shutdown), and stride-sample the middle to extract bounding box and feed/speed extremes — say in the Summary that you sampled.
2. **Detect the flavor.** Look for tell-tale codes: `M104/M109/M140/M190` (FFF 3D printer), `M3 S…` with `G1 Z` cuts (CNC mill or laser), `M6 T…` (tool change, CNC), `G28` style and comment headers (slicer name in `; generated by …` is decisive — PrusaSlicer/Cura/Bambu/OrcaSlicer/Simplify3D/SuperSlicer all stamp themselves). Note the detected flavor in the Summary.
3. **Parse modal state as you walk the file.** Track current units, positioning, WCS, feedrate, spindle/laser power, extruder temp, bed temp, current XYZ(E), and tool. Each `G0/G1/G2/G3` updates position; only commanded axes move. Honor `G90`/`G91` and `G92` (set position). Treat `G2/G3` as arcs (IJ or R form) — for the bounding box, approximate by sampling the arc.
4. **Compute the bounding box** from every commanded point after applying modal state. Report X/Y/Z min/max in the file's units.
5. **Group into phases.** Heuristics: a `M6`/`T…` line starts a new operation; long blocks of contiguous `G1` extrusion at constant Z are a layer (3D print); a `; LAYER:` or `;LAYER_CHANGE` or `; layer N` comment is decisive. Slicer object/feature comments (`;TYPE:WALL-OUTER`, `; FEATURE: …`) are gold — use them.
6. **Mine comments.** Lines starting with `;` (or text inside `()` on classic CNC) are author/slicer intent. Surface relevant comments verbatim in the narrative — they often state the goal better than the motion does.
7. **Flag risks.** Specifically check for: no `G28`/homing before motion, mixed G90/G91 without re-establishing, `G0` Z-down through likely stock height, `M3` without `S`, `M6` without a corresponding tool length offset, extruder/bed temperature wait codes (`M109/M190`) missing before extrusion, feedrate of 0, dwell `G4` with very long `P`/`S`.
8. **Write the document.** Follow the Output contract exactly. Use fenced code blocks only when quoting G-code; everything else is prose.

## Reference: codes worth recognizing on sight

Motion: `G0` rapid, `G1` linear feed, `G2`/`G3` CW/CCW arc, `G4` dwell, `G28` home, `G29` bed/auto-level, `G53` machine coords, `G54–G59` work offsets, `G90`/`G91` abs/rel, `G92` set position, `G17`/`G18`/`G19` plane, `G20`/`G21` inch/mm, `G94`/`G95` feed per min / per rev, `G81`/`G82`/`G83` drilling cycles, `G43` tool length offset.

M-codes (common): `M0` pause, `M1` optional stop, `M2`/`M30` end, `M3`/`M4`/`M5` spindle CW/CCW/stop, `M6` tool change, `M7`/`M8`/`M9` coolant, `M82`/`M83` extruder abs/rel, `M104`/`M109` hotend set / set+wait, `M106`/`M107` fan on/off, `M140`/`M190` bed set / set+wait, `M201`/`M203`/`M204`/`M205` limits, `M220`/`M221` feed/flow override.

Word letters in a block: `N` line number, `G` motion mode, `M` machine cmd, `X/Y/Z/A/B/C/U/V/W` axis target, `I/J/K` arc center offsets, `R` arc radius / retract, `F` feedrate, `S` spindle/laser power or temperature, `T` tool, `P/Q` cycle params, `E` extruder amount.

## Style rules

- Write for someone who can read G-code but doesn't want to. Stay concrete and quantitative.
- Use the file's units in your numbers; don't silently convert.
- Don't invent intent. If a move's purpose is ambiguous, say so.
- Keep the narrative section linear in file order — that's its job.
- If the file is huge and you sampled, **say so** in the Summary so the reader knows the bounding box and counts are estimates.
- Never modify the G-code unless the user asked for an annotated version; in that case, emit a copy with `;` comments inserted, never altering the original commands.
