# Compiling a C project with gcov coverage

When a task requires building a C project from source with gcov code-coverage
instrumentation and exposing the resulting binary on `PATH`, follow this
procedure.

## 1. Inspect the build system

Before running anything, identify the build harness:

- **autotools** (`./configure` + `Makefile.in` generated by `autoconf`):
  you can override `CFLAGS`/`LDFLAGS` from the environment.
- **autosetup** (tdbc, sqlite, etc. — `autosetup` + `Makefile`): also
  accepts environment `CFLAGS`/`LDFLAGS`; the bundled `configure` script
  picks them up before doing its own flag handling.
- **Plain `Makefile`**: edit the `CFLAGS`/`LDFLAGS` variables, or pass
  them on the make command line.

Look for `configure`, `autosetup`, `Makefile`, or `CMakeLists.txt` in the
source root and decide which path applies.

## 2. Extract sources into a clean working directory

```sh
mkdir -p <build-dir>
tar xzf <tarball> -C <build-dir> --strip-components=1
cd <build-dir>
```

`--strip-components=1` removes the leading versioned directory so paths
inside the build dir are stable.

## 3. Configure with coverage flags

For autotools/autosetup, pass the coverage flags as environment variables
*before* `./configure`:

```sh
CFLAGS='--coverage' LDFLAGS='--coverage' ./configure
```

`--coverage` is shorthand for `-fprofile-arcs -ftest-coverage` on gcc ≥ 4.8
and for `-fprofile-arcs -ftest-coverage` on clang. If the toolchain is
older, spell them out explicitly.

For a plain `Makefile`, do **not** invoke `./configure`; instead run
`make` with the flags appended:

```sh
make CFLAGS='--coverage' LDFLAGS='--coverage'
```

(only works if the Makefile uses the `CFLAGS` variable when compiling).

## 4. Build (parallel when possible)

```sh
make -j"$(nproc)"
```

The compiler emits a `.gcno` file next to each translation unit. Linking
pulls in `libgcov`. Do **not** strip the binary.

## 5. Install / expose on PATH

Pick the method that matches the task constraints:

- **Copy** the binary into a directory already in `PATH`
  (e.g. `/usr/local/bin`):
  ```sh
  install -m 0755 <binary> /usr/local/bin/
  ```
- **Symlink** instead of copy if you want the build dir to remain the
  source of truth:
  ```sh
  ln -sf "$(pwd)/<binary>" /usr/local/bin/<binary>
  ```
- **Prepend the build dir to `PATH`** for the current shell only:
  ```sh
  export PATH="$(pwd):$PATH"
  ```

## 6. Verify coverage is actually live

Before declaring success, confirm three things:

1. `gcov` (or `llvm-cov gcov`) is installed and runnable.
2. The installed binary still triggers coverage output:
   ```sh
   cd /tmp && rm -f *.gcda && <binary> <args> >/dev/null
   ls *.gcno *.gcda
   ```
3. `gcov <source-file>` produces a `.gcov` coverage report with non-zero
   line counts.

If `.gcno` files are missing after step 4, the configure step did not
honor the environment flags — re-check the build system type and the
flag names (`--coverage` vs `-fprofile-arcs -ftest-coverage`).

## Common pitfalls

- Running `./configure` *after* exporting `CFLAGS` is required; some
  scripts reset flags internally — re-export right before invocation.
- Coverage builds need the runtime support library linked; do not
  override `LIBS` to an empty value.
- If a `Makefile` hard-codes `CFLAGS=`, edit it or pass an override on
  the command line, not via the environment.
- Stripping the binary removes `.gcno` metadata; skip `strip`.
